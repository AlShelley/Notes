双因素覆盖不全

这篇文章引用自【呆子不开口】在SDCC上的关于账号通行证安全相关的PPT。膜拜大佬

漏洞原理

双因素后获取的凭证，在无需双因素处可以获得。

常见的场景有：

1、web版登陆需要双因素认证，但app端却没跟上仍然可以用账号密码登陆
2、在登陆处需要，在某些特殊功能处却可以通过特殊白名单参数绕过
3、某些双因素认证后才能获得的凭证，可以在普通权限下通过某接口中获得

案例：

  • 淘宝异地登录短信认证绕过
  • 某微博的双因素认证绕过
  • 某app移动端登录获取pc端双因素后的cookie

修复方案

双因素认证后需要额外设置新的凭证
并保证此凭证不在其他接口输出

使用GET方法传输用户账号密码的危害

通过 Get 方式提交数据时，用户名和密码将出现在 URL 上，如果页面可以被缓存（浏览器缓存、cdn 缓存、服务器缓存、中间件日志缓存等）或者其他人可以访问客户这台机器，就可以从历史记录获得该用户的帐号和密码，所以表单提交建议使用 Post 方法。

验证码安全

由于业务迭代频繁，产品线众多...好了不废话了。简单来说就是很多网站的登录界面没有验证码，只有账号密码输入框，这样不安全。

什么是验证码

全自动区分计算机和人类的图灵测试（Completely Automated Public Turing test to tell Computers and Humans Apart，简称CAPTCHA），俗称验证码，是一种区分用户是计算机或人的全自动化程序。

验证码种类

1、图形验证码
2、滑动验证码（比如极验）
3、答题验证码
4、手机短信/邮箱验证码
5、动态口令

验证码的作用

人机识别：识别使用对应功能的是人还是机器。
防止机器暴力攻击：防止攻击者在短时间内用机器进行批量操作（常见功能场景如：登录）。

验证码工作原理

验证码常用场景与作用

登录接口常使用验证码来防御攻击者的撞库攻击
注册接口常用验证码来防御批量恶意注册
身份认证
重要表单提交
修改密码
支付等功能

验证码使用

重要接口要加验证码
常见的如登录、注册等接口推荐使用滑动验证码；重要的如改密、支付等功能推荐使用手机短信验证码；vpn登录、身份认证等推荐使用动态口令。

验证码对抗技术

1、OCR 识别图形验证码
2、机器识别验证码（常见流程：去噪处理-图像二值化-切片处理-文件图片标准化-字符模板识别）
3、LTE／4G 伪基站＋GSM 中间人攻击实现手机短信验证码攻击（相关安全事件可见"这下一无所有了"：http://netsecurity.51cto.com/art/201808/580812.htm）
4、验证码流程绕过（空验证码、返回包含验证码、验证码固定、验证码前端校验、验证流程有误）
5、人工打码平台

最近遇到的一个靶机

1、查看 http 流量，后台扫描等进行信息收集，得出其用的是 xxxxcms，然后百度进一步进行信息收集，得出该 cms 存在过 sql 注入、文件包含等漏洞。
2、全站测试 sql 注入，发现确实存在注入点
3、上 sqlmap 进行扫描，得出管理员账号密码，得出 mysql 为普通用户权限。
4、登录网站后台，利用文件包含漏洞获取部分.php 文件信息，对其进行编辑写入小马。（这块之前没怎么玩过，因此稍微花费了一点时间，目的达到了，不过不优雅）
5、通过小马传入大马，然后连接大马。
6、提权拿 linux 的 root，这里我不太懂，因此正在找资料，希望今晚能搞定提权这事。

提权这块研究一下，搞出来了再写。目前的思路是通过 shell 查看操作系统版本和内核版本，查看已安装的包和程序，看看有没有一直漏洞可以利用。
